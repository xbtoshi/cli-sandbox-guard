## Independent published-bundle verification

After publication, all five release assets were downloaded into a fresh host temporary
directory and checked with:

```text
LC_ALL=C scripts/verify-release-artifacts.sh published 0.3.0-alpha.3 <download-directory>
```

The verifier accepted the exact five-asset set, recomputed the archives and their inner-file
hashes and sizes, checked binary formats and executable modes, verified the shared Linux ARM64
guest helper identity, and found that `SHA256SUMS` and `manifest.json` matched the recomputed
data. The published `SHA256SUMS` also matches the hashes printed in the `verify` job log of the
tag-triggered run, closing the chain from CI-tested binaries to published assets.

```text

9bdf0bd9ca64e4246670d6bd4b8e6ebbcbc82bf0ad626d42d8b8dd1999e5658c  sandbox-guard-0.3.0-alpha.3-linux-aarch64.tar.gz
df34b1ba6778b45ed1f0aa411a74edca59f63647805a980cac16b841601054d9  sandbox-guard-0.3.0-alpha.3-macos-arm64.tar.gz
```

**Verifier caveat discovered during this validation:** `scripts/verify-release-artifacts.sh`
builds the manifest `files` arrays in `sort` order, and `sort` collation is locale-sensitive.
On a host locale such as `en_HK.UTF-8` the recomputed array order differs from the CI-generated
manifest (`LICENSE` sorts after `guard*`), and published-mode verification fails with
`manifest.json does not match the recomputed artifact manifest` even though every hash and size
matches. Run the verifier under `LC_ALL=C` until the script pins its own collation. This is a
verifier portability defect, not an artifact integrity problem.

## Live Lima validation

The published macOS ARM64 archive was extracted without installing it over the local
development build. Its packaged Linux ARM64 `lima-guest/guard-helper`
(`9311d317c87d72f51056a9cfa227b8918fed9a4f54bb276f88dce44e9843d33d`, matching
`manifest.json`) was copied to a unique temporary path under `/usr/local/bin` in the existing
mountless `sandbox-guard` Lima instance and its in-guest SHA-256 re-verified before use. The
packaged macOS `guard` binary was then run with:

```text
guard test --backend macos-lima --require-cgroup --helper <temporary-guest-helper>
```

All probes passed against the published binaries: filesystem boundary, environment boundary,
PID namespace, host loopback isolation, controlled egress denial and direct-bypass isolation,
seccomp namespace and process-memory denial, seccomp thread compatibility, trusted supervisor
memory, trusted preflight sequencing, rlimits, and required cgroup v2 enforcement (exit 0).
The temporary guest helper and upload staging file were removed afterwards and their removal
confirmed. An equivalent probe had also been run against the same packaged binaries during
release execution, before the draft was published.
